# Ticket Inbox Design (Sub-project F)

**Status:** design approved by user on 2026-04-18. Ready for plan + implementation.

## 1. Goal

Turn the dashboard's current "plain table of reports" into a working triage inbox. Admins and developers can filter, search, assign, prioritize, tag, and transition status on incoming feedback reports, with a small auto-recorded activity log that answers "why was this closed?" without building a full comments system.

## 2. Scope — locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| **Status states** | Fixed 4: `open` / `in_progress` / `resolved` / `closed` | Matches CLAUDE.md §3.5; covers "someone's on it" signal without per-project config overhead. |
| **Assignment** | `assignee_id` FK to `user.id` (`text`, better-auth convention). Picker scoped to `developer` / `owner` members of the project. Viewers cannot be assignees. | Viewers are read-only triagers; no point assigning work they can't action. |
| **Categorization** | Both `priority` (fixed enum `low` / `normal` / `high` / `urgent`) AND `tags` (`text[]`, free-form, DISTINCT-unnest derives project's tag list). | Priority is the "what do I work on next" signal. Tags are the "which system is this" signal. Both fit naturally into the report row. |
| **Filter/search UX** | Status tabs across the top + text search + sort dropdown + collapsible faceted sidebar (Assignee / Priority / Tags). Filter state in URL query string. | Covers real triage workflows. Filter state in URL = free "saved views" via bookmarks. |
| **Collaboration** | Activity log only (auto-recorded events from every mutation). No user comments in v1. | 80% of "why was this closed?" answered without building a comments editor, mention resolution, or email. |
| **Bulk operations** | Bulk status + bulk assignee only. Checkbox column + sticky bulk action bar. | Covers 95% of real batching (close spam, assign a batch). Bulk priority/tags are case-by-case edits — leave for v2. |
| **Real-time** | TanStack Query with `refetchOnWindowFocus: true` + 15s background poll. No websockets/SSE. | Self-hosted single-workspace admin tool; the cost of a socket stack exceeds the benefit of sub-15s freshness. |
| **Permissions** | `viewer` reads, `developer`+ mutates. Enforced at the route via `requireProjectRole`. | Consistent with existing patterns in A/B. |

## 3. Out of scope for v1

Flagged so they can be pulled in later without re-designing:

- Keyboard shortcuts (j/k row nav, e → in_progress, y → resolved, / → search). Add once real users triage daily.
- Saved filter presets beyond URL bookmarks.
- Report deletion UI. Still possible via project cascade.
- Email notifications (on assignment, on status change, etc.). Requires the Resend/SMTP stack — out of this sub-project.
- Status reason / resolution note. Add alongside comments in a future collab pass.
- Full-text search with `pg_trgm` + GIN indexes. v1 uses ILIKE, which is fine up to ~10k reports per project. Flagged in the threat model.
- Custom per-project statuses. DB columns are check-constrained `text` (not native `ENUM`), so adding future values doesn't require an `ALTER TYPE` migration.

## 4. Architecture

### 4.1 Approach

**Extend in place.** Add triage columns to the existing `reports` table; add one `report_events` table for the activity log; extend the existing `GET /api/projects/:id/reports` endpoint with query params + facet counts; add two new mutation endpoints; replace the current `reports.vue` table with a faceted three-column inbox layout reusing D's drawer.

Alternatives considered and rejected:

- **Separate inbox vs archive.** Double the surface area for no benefit — this is a single-workspace app, not a product with two distinct user journeys.
- **CQRS read model** (materialized `report_inbox_view` via triggers). Premature optimization; v1 scale is <10k reports per project.

## 5. Data model

### 5.1 `reports` extensions

Add four columns to the existing table. All have defaults so existing rows remain valid (migrate-in-place, no backfill).

| column | type | default | notes |
|---|---|---|---|
| `status` | `text` check-constrained to `open` / `in_progress` / `resolved` / `closed` | `'open'` | |
| `assignee_id` | `text` nullable, FK → `user.id` ON DELETE SET NULL | `null` | better-auth user IDs are text, not uuid |
| `priority` | `text` check-constrained to `low` / `normal` / `high` / `urgent` | `'normal'` | |
| `tags` | `text[]` | `'{}'` | DISTINCT-unnest derives project-level tag suggestion list |
| `updated_at` | `timestamptz` | `now()` | bumped on every triage mutation; enables "recently triaged" sort |

Check constraints live on the column (`CHECK (status IN (...))`), not via native pg `ENUM`, so future additions don't require `ALTER TYPE`.

### 5.2 New table: `report_events`

| column | type | notes |
|---|---|---|
| `id` | `uuid` pk default `gen_random_uuid()` | |
| `report_id` | `uuid` FK → `reports.id` ON DELETE CASCADE | cascade with the report |
| `actor_id` | `text` nullable FK → `user.id` ON DELETE SET NULL | `null` = system-emitted event |
| `kind` | `text` check-constrained to `status_changed` / `assignee_changed` / `priority_changed` / `tag_added` / `tag_removed` | |
| `payload` | `jsonb` | `{ from, to }` for scalar changes; `{ name }` for tag events |
| `created_at` | `timestamptz` default `now()` | |

### 5.3 Indexes

```sql
CREATE INDEX reports_project_status_created_idx ON reports (project_id, status, created_at DESC);
CREATE INDEX reports_project_assignee_idx       ON reports (project_id, assignee_id);
CREATE INDEX reports_project_priority_idx       ON reports (project_id, priority);
CREATE INDEX reports_tags_gin_idx               ON reports USING GIN (tags);
CREATE INDEX report_events_report_created_idx   ON report_events (report_id, created_at DESC);
```

Existing `reports_project_created_idx` stays — covers the "all reports newest first" sort without a status filter.

### 5.4 DTO shape

`ReportSummaryDTO` (extends existing — additive, no removals):

```ts
{
  id: string
  title: string
  description: string | null
  context: ReportContext  // from D
  reporterEmail: string | null
  pageUrl: string
  thumbnailUrl: string | null
  receivedAt: string        // was createdAt; preserved for back-compat
  updatedAt: string         // new

  status: 'open' | 'in_progress' | 'resolved' | 'closed'
  priority: 'low' | 'normal' | 'high' | 'urgent'
  tags: string[]
  assignee: null | { id: string; name: string | null; email: string }
}
```

`ReportEventDTO`:

```ts
{
  id: string
  createdAt: string
  actor: null | { id: string; name: string | null; email: string }
  kind: 'status_changed' | 'assignee_changed' | 'priority_changed' | 'tag_added' | 'tag_removed'
  payload:
    | { from: string; to: string }
    | { from: { id, name, email } | null; to: { id, name, email } | null }
    | { name: string }
}
```

## 6. API

### 6.1 Extend `GET /api/projects/:id/reports` (viewer+)

Query params (all optional):

| param | shape | default |
|---|---|---|
| `status` | CSV of enum values | unfiltered |
| `assignee` | `me` / `unassigned` / user UUID, CSV | unfiltered |
| `priority` | CSV of enum values | unfiltered |
| `tag` | CSV of tag strings, **AND** semantics (all must match) | unfiltered |
| `q` | search string, ILIKE against title OR description, case-insensitive | unset |
| `sort` | `newest` / `oldest` / `priority` / `updated` | `newest` |
| `limit` | 1–100 | 50 |
| `offset` | ≥0 | 0 |

**Response:**

```ts
{
  items: ReportSummaryDTO[]
  total: number
  facets: {
    status:    Record<'open'|'in_progress'|'resolved'|'closed', number>
    priority:  Record<'low'|'normal'|'high'|'urgent', number>
    assignees: Array<{ id: string | null; name: string | null; email: string | null; count: number }>
    tags:      Array<{ name: string; count: number }>  // top 20 by count, desc
  }
}
```

**Facet count semantics:** all counts use the *current* filter set (same as the `items` query). Predictable — what's in the list matches the bucket counts. Future iteration can shift to "count ignores its own dimension" if triage workflows demand.

**Limits (DoS mitigation):** each CSV param accepts ≤10 values; `q` ≤200 chars; `limit` hard-capped at 100.

### 6.2 `PATCH /api/projects/:id/reports/:reportId` (developer+)

**Body** (all fields optional — partial update):

```ts
{
  status?: 'open' | 'in_progress' | 'resolved' | 'closed'
  assigneeId?: string | null         // null = unassign
  priority?: 'low' | 'normal' | 'high' | 'urgent'
  tags?: string[]                    // full replacement; server diffs against current
}
```

**Server semantics:**

1. Validate body via Zod; reject unknown keys.
2. If `assigneeId` present, verify the user is a `developer`+ member of *this* project. Else 400 "not a mutable member".
3. Open a single DB transaction:
   - `UPDATE reports SET ...` for changed fields, bumping `updated_at`.
   - `INSERT INTO report_events` one row per *changed* field:
     - `status` changed → 1 event `status_changed { from, to }`
     - `assigneeId` changed → 1 event `assignee_changed { from: {id,name,email}|null, to: {id,name,email}|null }` (server resolves user records for both sides)
     - `priority` changed → 1 event `priority_changed { from, to }`
     - `tags` changed → diff against current: one `tag_added { name }` per added, one `tag_removed { name }` per removed. **Not** a single "tags replaced" event.
   - If no fields actually differ from current values, no events emitted, no `updated_at` bump, 200 returns the unchanged DTO.
4. Return the updated `ReportSummaryDTO`.

### 6.3 `POST /api/projects/:id/reports/bulk-update` (developer+)

**Body:**

```ts
{
  reportIds: string[]                // 1–100 UUIDs, all belonging to :id
  status?: 'open' | 'in_progress' | 'resolved' | 'closed'
  assigneeId?: string | null
}
```

Requires at least one of `status` / `assigneeId`. `priority` / `tags` not supported in bulk for v1.

**Server semantics:**

- Validate all `reportIds` belong to the project (single query). Any stranger → 400, nothing written.
- Single transaction: per-report update + per-changed-field event emission, exactly like the single PATCH path (shared emit helper in `server/lib/report-events.ts`).
- Return `{ updated: string[] }` — the IDs whose rows actually changed. Reports already in the target state skip the update and aren't included.

### 6.4 `GET /api/projects/:id/reports/:reportId/events` (viewer+)

Paginated event feed for one ticket.

**Query:** `limit` (1–100, default 50), `offset` (≥0, default 0).

**Response:** `{ items: ReportEventDTO[], total: number }`. Sorted `created_at DESC`.

Events carry the actor embedded at request time — if the user has since been deleted, `actor` becomes `null` (the FK's ON DELETE SET NULL leaves historic events intact).

### 6.5 Extend `GET /api/projects/:id/members` (viewer+)

Add optional query param `role` = CSV of `owner` / `developer` / `viewer`. Used by the assignee picker to show only developer+ members. Already role-scoped at the route — just a projection.

**Row anatomy:** checkbox · priority badge · title · assignee avatar (or `—`) · relative time. Row click (outside checkbox) opens the drawer.

**URL encoding.** Every filter/sort/selection-less state goes in the query string: `?status=open&priority=high&assignee=me&tag=mobile&q=login&sort=newest`. Shift-click on facets toggles multi-select.

**Bulk bar** slides up from the bottom when at least one row is checked. Shows count + two pickers + Clear. Selection persists across pagination until cleared.

**Empty states:**

- Filters empty the list: "No reports match these filters" + [Clear filters] button.
- Project has zero reports: existing empty-state card with embed snippet unchanged.

**Loading:** skeleton rows in the list; spinner in the drawer when it first opens.

### 7.2 Drawer changes

Building on D's `Overview / Console / Network / Cookies` tab shell:

- **Triage panel pinned above the tab bar** (new). Four inline-editable controls, each a popover:
  - Status pill
  - Assignee pill (searchable member picker, "Unassign", "Assign to me")
  - Priority pill (with color swatches)
  - Tag chips row (type-ahead autocomplete from project tags + `x` to remove)
- Each change optimistically updates the UI, fires `PATCH .../reports/:reportId` with just the delta, invalidates the list query on success, and rolls back on error.
- **New `Activity` tab inserted *before* Overview.** Reverse-chrono event feed. Each row: actor avatar + imperative-past-tense sentence ("Alex changed status Open → In progress") + relative timestamp. When I open a report to triage, Activity is the first thing I see; when I open to diagnose, I click Overview.

### 7.3 Permissions

- `viewer`: list + drawer read-only. Triage pills are rendered as disabled badges (not buttons). Bulk checkboxes hidden.
- `developer` / `owner`: full mutate.

### 7.4 Live refresh

List query and drawer-events query both use TanStack Query with:

```ts
{ refetchOnWindowFocus: true, refetchInterval: 15_000 }
```

If User A sees User B's change, it happens within 15s or instantly on tab focus. Optimistic updates make local mutations feel instant.

## 9. Threat model additions

To be appended to `docs/superpowers/security/threat-model.md` during implementation:

- **Mutation authorization:** every PATCH / bulk-update goes through `requireProjectRole(event, id, 'developer')`. Tested.
- **Assignee scoping:** the picker and validation constrain assignees to developer+ members of the *same* project. A user ID from another project's membership returns 400. Tested.
- **Concurrent writes:** last-write-wins on PATCH. Two admins racing to change status = whichever hits the DB second wins, and the events log shows both transitions in order. No locking.
- **Event log integrity:** mutation + event inserts share a single DB transaction. Either both land or neither — no ghost events.
- **Search performance:** ILIKE without a trigram index is acceptable up to ~10k reports per project. Follow-up: add `pg_trgm` + GIN on `lower(title || ' ' || coalesce(description, ''))` once real installs need it.
- **DoS mitigation:** query param arrays capped at 10 values per key; `q` ≤200 chars; bulk `reportIds` ≤100; `limit` ≤100.
